# Refactoring Task: NanoLoc 2.0 - Clean Architecture, Auth Simplification & Docker Auto-Healing

We are doing a major refactoring for the `2.0` branch. The target audience is a small internal team (around 10 people). We need to achieve 4 architectural goals: fix Docker permissions permanently, simplify Auth, remove strict RBAC, and introduce a clean Service Layer.

Please implement the following architectural changes:

## Objective 1: Clean Architecture & Modularization (Service Layer)
**Goal:** Decouple business logic from Next.js API route handlers to make the codebase maintainable and extensible.
- **Service Layer (`src/lib/services/`):** Create this directory and abstract the core logic into specific service classes/files:
  - `project.service.ts`: Handle all Prisma CRUD operations for Projects, TranslationKeys, and TranslationValues.
  - `translate.service.ts`: Encapsulate AI logic, prompt construction, and batch processing flow.
  - `storage.service.ts`: Abstract the file processing logic. Move the XML parsing (Import) and XML string building (Export) here.
- **Thin API Routes (`src/app/api/...`):** API routes must become thin controllers. They should only: 
  1. Verify the session.
  2. Parse and validate the request body/params.
  3. Call the appropriate Service method.
  4. Return the standard JSON response.

## Objective 2: Docker Permission Auto-Healing (Root to su-exec)
**Goal:** Ensure the SQLite database is always writable, bypassing host-machine volume permission conflicts.
- **Dockerfile:** Ensure the container starts as `root` (remove `USER nextjs`). Install `su-exec` and `openssl`. Ensure CRLF is stripped (`sed -i 's/\r$//'`) from the startup script.
- **Startup Script (`scripts/start.sh`):** Intercept the boot process. As root, explicitly run `chown -R nextjs:nodejs /app/prisma/data` and `chmod -R 775 /app/prisma/data`. Then, use `su-exec nextjs` to drop privileges and run `npx prisma migrate deploy` and `node server.js`.
- **Compose:** Standardize the database mount exactly to `/app/prisma/data`.

## Objective 3: Minimalist Authentication (Credentials Only)
**Goal:** Replace heavy NextAuth configurations with a simple, robust Email/Password flow.
- **Auth Config:** Configure NextAuth to use *only* the `CredentialsProvider`.
- **Validation:** Implement standard email lookup and bcrypt password verification in the `authorize` callback. Use the `jwt` session strategy.

## Objective 4: Team-wide Visibility (Remove RBAC)
**Goal:** Eliminate project-level user access restrictions.
- **Global Access:** Audit all project-related APIs and Service methods. Remove all logic filtering projects by the current user's email/ID. 
- **Rule:** If a user is authenticated (valid session), they can view, edit, import, and export ALL projects in the database.

## Constraints
- Do not break existing API response schemas so the frontend UI remains unbroken.
- Maintain the Audit feature (recording `lastModifiedByUserId` when updating terms), but it should rely on the basic global session.
- Ensure TypeScript compiles successfully after the modularization.
